# Project-specific instructions

## Tech Stack

- Python 3.12+
- Telegram API (Telethon/Pyrogram)
- MCP Protocol

## Code Style

- line-length = 99
- Следовать стилю существующего кода
- Type hints обязательны

## Module Naming

```text
Папка:  src/<domain>/
Файл:   <сущность>.py
Класс:  <Сущность>
```

## Project Structure

```
src/
├── telegram/      # Telegram API client
├── mcp/           # MCP server implementation
├── search/        # Search functionality
└── models/        # Pydantic models
```

## Git

- Работаем в main
- Pre-commit hook: sdd_validator.py + pytest-check

## Testing

- Spec-тесты: `tests/spec/<domain>/test_<component>_spec.py`
- Запуск: `pytest tests/spec/ -v`

## Domain Context

MCP сервер для поиска по Telegram чатам.
Основные фичи: индексация истории, полнотекстовый поиск.

## Red Flags (самопроверка перед отправкой)

**Если видишь в своём ответе — переделай:**

1. **Вопросы типа "Что ещё поправить?"** — пиши качественно сразу, не перекладывай ревью на пользователя
2. **Просьбы сделать то, что можешь сам** — "CV по запросу" когда пользователь сказал что приложит CV
3. **Факты не из источника** — не выдумывай опыт/навыки/цифры, бери только из данных
3. **Маркетинговый буллшит** — "Strong focus on...", "plays a vital role", "remarkable" — удалить
4. **Смайлики/эмодзи** — AI паттерн, не использовать
5. **Неполная информация** — если данные есть, используй их полностью
6. **Непроверенные расчёты** — даты, сроки, суммы — перепроверь перед написанием
7. **Непроактивность** — не заставляй пользователя просить добавить очевидные вещи по одной
8. **Перезапись пользовательских данных** — ВСЕГДА читай заметку перед обновлением и сохраняй пользовательские правки (комментарии, статусы). Используй `append_section` или точечные правки вместо полной перезаписи

**Принцип:** Сообщение должно быть таким, что ты всё учёл и всё проверил. Первый драфт = финальный.
