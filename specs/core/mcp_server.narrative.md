# MCP Server

## Intent

Точка входа для Claude в Telegram-данные. MCP сервер оркестрирует все остальные модули (telegram_client, storage, search) и предоставляет Claude набор tools для работы с чатами.

Claude никогда не обращается к Telegram API напрямую — только через локальные файлы. Это гарантирует скорость и предсказуемость. Если данных нет — автоматически запускается синхронизация (lazy load), либо Claude может запросить sync явно.

MCP может использоваться одновременно из нескольких проектов, поэтому операции синхронизации защищены lock'ом на уровне чата.

Ответы tools оптимизированы под Claude: структурированные данные в JSON, контент сообщений в markdown. Лимиты по символам (8000 default / 20000 max) защищают от переполнения контекста.

Ошибки возвращаются в понятном формате с подсказками, не как exceptions

## Clarifications

### Lazy load поведение

Q1: Когда триггерить автоматическую синхронизацию?
- При любом запросе к чату без данных
- Только при явном sync_chat вызове

Предлагаю: автоматически при первом запросе к чату (lazy load). Если данные есть — читаем из файлов.

A1: Автоматиески + по запросу явному (2 варианта)

### Валидация параметров

Q2: Как обрабатывать невалидные параметры?
- chat_id не в whitelist
- Несуществующий block
- msg_id не найден

Предлагаю возвращать понятные ошибки в content, не exceptions:
```
{"error": "Chat 123 not in whitelist", "hint": "Use list_chats to see available chats"}
```

A2: Ок

### Формат ответов tools

Q3: Какой формат ответов?
- Чистый текст (markdown)
- JSON
- Смешанный (JSON для list_chats, markdown для read_*)

Предлагаю:
- list_chats, search_blocks → JSON (структурированные данные)
- read_* → markdown (для чтения Claude)
- sync_chat → JSON (статус)

A3: Ок

### Лимиты и пагинация

Q4: Как работать с большими результатами?
- search_blocks limit=20 — достаточно?
- read_blocks — максимум блоков за раз?
- read_recent limit=50 — достаточно?

Предлагаю:
- search_blocks: limit 1-100, default 20
- read_blocks: max 10 блоков
- read_recent: limit 1-200, default 50

A4: Ок. Старайся везде по минимуму. Если что, потом увеличим, если ты поймешь, что стоит увеличить.

### Обработка ошибок Telegram

Q5: Что делать при ошибках Telegram API (при sync)?
- FloodWait — ждать или сразу ошибка?
- Network errors — retry?

Предлагаю: при FloodWait возвращать ошибку с временем ожидания. Retry на network errors — 3 попытки.

A5: Ок

### Конкурентные запросы

Q6: Может ли Claude делать несколько запросов параллельно?
- Если да, нужна ли защита от race conditions?
- Особенно при sync_chat

Предлагаю: sync_chat — только один на чат одновременно (lock). Остальные tools — без ограничений.

A6: Могу использовать этот MCP одновременно в нескольких проектах

Q6.1: Для защиты sync_chat от параллельного запуска из разных процессов нужен file-based lock (например, `data/chats/<chat_id>/.sync.lock`). In-memory lock не поможет между процессами. Согласен?

A6.1: Ок
